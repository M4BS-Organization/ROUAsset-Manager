## 2. スキーマ移行

### 2.1 完了済み作業

Accessスキーマ解析（Masked_Schema_M7.accdb, Masked_Logic_M7.accdb）に基づき、421テーブルのPostgreSQL作成スクリプトを生成済み。ローカル環境にm7_dbを構築し、全テーブル・制約の作成が完了している。

### 2.2 テスト環境のDB設定

**テスト環境では `lease_m4bs_test` を使用すること。** `DbConnectionManager.vb` のデフォルトDB名は本番用の `lease_m4bs` であるため、テスト実行時に誤って本番DBに接続するリスクがある。テスト環境での接続切り替え手順は以下の通り:

1. `App.config` の `LeaseM4BS_Test` 接続文字列を使用:
   ```xml
   <add name="LeaseM4BS_Test" connectionString="Host=localhost;Port=5432;Database=lease_m4bs_test;Username=lease_m4bs_user;Password=...;" />
   ```
2. `LeaseM4BS.Tests` プロジェクトの `TestInitialize` で `LeaseM4BS_Test` 接続文字列を明示的に指定
3. 本番DB（`lease_m4bs`）には絶対に接続しないこと

### 2.3 データ型マッピング

| Access型 | PostgreSQL型 | VB.NET型 | 注意事項 |
|---|---|---|---|
| AutoNumber | SERIAL / BIGSERIAL | Integer / Long | 自動採番。シーケンス値の引継ぎに注意 |
| Text(n) | VARCHAR(n) | String | n=255がAccessデフォルト |
| Long Integer | INTEGER | Integer | 4バイト整数 |
| Double | DOUBLE PRECISION | Double | 浮動小数点の精度差に注意 |
| Currency | NUMERIC(19,4) | Decimal | 金額項目。精度を保証 |
| Date/Time | TIMESTAMP | DateTime | タイムゾーンの取扱いに注意 |
| Yes/No | BOOLEAN | Boolean | Access: -1/0, PostgreSQL: true/false |
| Memo | TEXT | String | 可変長。文字コード変換に注意 |
| OLE Object | BYTEA | Byte() | バイナリデータ |

### 2.4 特記事項

- AutoNumberフィールドはSERIAL型に変換。移行後にシーケンスの現在値を移行データの最大値+1に設定する必要がある
- Yes/NoフィールドはAccessでは-1(True)/0(False)だが、PostgreSQLではtrue/false。変換スクリプトで対応
- Date/TimeフィールドはAccessのOLEオートメーション日付からPostgreSQLのTIMESTAMPへ変換
- 文字コードはAccessのShift_JISからPostgreSQLのUTF-8へ変換

## 3. データ移行

### 3.1 移行手順

```
フェーズ1: 準備
├── Step 1: Accessデータのエクスポート（CSV/SQL形式）
├── Step 2: 文字コード変換（Shift_JIS → UTF-8）
├── Step 3: データ型変換スクリプト作成
└── Step 4: テスト環境（m7_db_test）でのリハーサル

フェーズ2: 移行実行
├── Step 5: PostgreSQLテーブルのTRUNCATE（既存データクリア）
├── Step 6: 外部キー制約の一時無効化
├── Step 7: COPYコマンドによるバルクインサート
├── Step 8: シーケンス値の再設定
├── Step 9: 外部キー制約の再有効化
└── Step 10: データ整合性チェック

フェーズ3: 検証
├── Step 11: レコード件数の比較検証
├── Step 12: サンプルデータの目視確認
├── Step 13: 集計値の突合検証
└── Step 14: 移行完了報告
```

### 3.2 移行スクリプト例

```sql
-- Step 5: テーブルクリア
TRUNCATE TABLE tm_ssuw CASCADE;

-- Step 6: 外部キー制約の一時無効化
SET session_replication_role = 'replica';

-- Step 7: CSVからのバルクインサート
COPY tm_ssuw FROM '/path/to/tm_ssuw.csv'
  WITH (FORMAT csv, HEADER true, ENCODING 'UTF8');

-- Step 8: シーケンス値の再設定
SELECT setval('tm_ssuw_id_seq', (SELECT MAX(id) FROM tm_ssuw));

-- 【推奨】シーケンス名の特定には pg_get_serial_sequence() を使用
-- PostgreSQLのSERIAL型は {テーブル名}_{カラム名}_seq の命名規則に従う
-- 例: tm_ssuw テーブルの id カラムのシーケンス名を取得
SELECT pg_get_serial_sequence('tm_ssuw', 'id');
-- → 'public.tm_ssuw_id_seq' が返される

-- シーケンス値を現在の最大値に合わせて更新（汎用的な書き方）
SELECT setval(pg_get_serial_sequence('tm_ssuw', 'id'), (SELECT MAX(id) FROM tm_ssuw));

-- Step 9: 外部キー制約の再有効化
SET session_replication_role = 'origin';

-- Step 10: 件数確認
SELECT COUNT(*) FROM tm_ssuw;
```

## 4. ロジック移行

### 4.1 DAO → VB.NET 移行パターン

| DAO（VBA） | VB.NET（Npgsql） | 備考 |
|---|---|---|
| DAO.Recordset | DataTable + CrudHelper.ExecuteSelect() | ループ処理をFor Eachで置換 |
| DoCmd.RunSQL | CrudHelper.ExecuteNonQuery() | パラメータ化クエリ使用 |
| CurrentDb.Execute | CrudHelper.ExecuteNonQuery() | トランザクション対応 |
| BeginTrans/CommitTrans | CrudHelper.ExecuteTransaction() | ロールバック自動対応 |
| Err.Number | DataAccessErrorCode | SQLSTATE→独自コードマッピング |
| Errors コレクション | DataErrorCollection | DAO互換の設計 |

### 4.2 フォーム移行計画

160フォーム（VBAコード5.4MB）を段階的に移行。

| 分類 | 割合 | 移行優先度 | 方針 |

| フレックスリスト | 約30% | Phase 5-10 | FlexListForm共通基盤で対応 |
| マスタ管理 | 約25% | Phase 12+ | Entity-Repositoryパターン活用 |
| 検索ダイアログ | 約15% | Phase 7 | FlexList検索機能に統合 |
| レポート | 約15% | Phase 12+ | 帳票出力機能として実装 |
| その他 | 約15% | Phase 12+ | 個別対応 |

## 5. 検証計画

### 5.1 データ検証項目

| # | 検証項目 | 方法 | 合格基準 |
|---|---|---|---|
| 1 | レコード件数 | Access vs PostgreSQL の COUNT比較 | 全テーブルで完全一致 |
| 2 | 主キー整合性 | PK重複チェック | 重複0件 |
| 3 | 外部キー整合性 | FK制約の有効化成功 | エラー0件 |
| 4 | データ型精度 | Currency/Double型のサンプル比較 | 小数点以下4桁まで一致 |

| 6 | 日付データ | 日付フィールドのサンプル比較 | 完全一致 |
| 7 | Boolean変換 | Yes/No → BOOLEAN のサンプル確認 | 全件正確に変換 |

## 8. リスクと対策

| リスク | 影響度 | 対策 |
|---|---|---|
| データ型変換での精度損失 | 高 | Currency→NUMERIC(19,4)で精度保証。変換前後の集計値を突合 |
| 文字コード変換の文字化け | 高 | Shift_JIS→UTF-8変換スクリプトを事前テスト。全角/半角混在パターンを検証 |
| AutoNumber→SERIAL移行でのID競合 | 中 | シーケンス値をMAX(id)+1に再設定 |
| 大規模テーブルの移行タイムアウト | 中 | バッチ分割（1000行単位）で対応 |
| OLE ObjectデータのBYTEA変換失敗 | 低 | 該当データの有無を事前調査。必要に応じて個別対応 |
| 認証情報の流出（ハードコードパスワード） | 高 | `DbConnectionManager.vb` のデフォルト接続文字列に開発用パスワード（`iltex_mega_pass_m4`）がハードコードされている。本番環境では必ずApp.configまたは環境変数の接続文字列で上書きすること。gitの履歴にも平文で残るため、本番パスワードは別途管理すること。 |
