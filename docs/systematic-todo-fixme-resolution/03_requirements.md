# 要件定義書: TODO/FIXME体系的解消

## 1. 目的とスコープ

Access版 LeaseM4BS を VB.NET/WinForms へマイグレーションする過程で、コードベース内に散在するTODO/FIXME・「不明」コメントを体系的に解消し、業務要件を満たす完全な実装を実現する。

## 6. リスクと前提条件

### リスク
| リスク | 影響度 | 対応方針 |
|---|---|---|
| Access版 .mdb がバイナリのためVBA参照不可 | 高 | エクスポートツール（Acess MDB Decompiler等）使用または推測ロジックで実装し仮定として明記 |
| f_CHUKI_RECALC の一括更新が本番データに影響 | 高 | 実行前に必ずバックアップ取得を促すUIガイドを追加 |
| グレーアウト条件が画面仕様に依存 | 中 | Form_f_flx_KHIYOの実装パターンを流用し、不一致はIssueとして残す |
| 削除処理の参照整合性エラー | 中 | 子テーブルの削除順序（d_haif → d_kykm → d_kykh等）を正確に定義する |

### 前提条件
- PostgreSQL DBスキーマ（d_kykm, d_kykh, d_haif, d_henf, d_gson等）は変更しない
- Access版VBAコードは .mdb バイナリのみ存在し、テキスト形式での直接参照は不可
- 計算ロジックの不明箇所は「Access版と同じ動作と推定」として実装し、仮定事項に記載する

## 8. 仮定事項

以下の事項はAccess版VBAが直接参照できないため、既存VB.NETコードの文脈と業務ロジックから推定した仮定事項である。実装前に担当者の確認が必要な場合はその旨を記載。

1. **f_flx_BUKN の保守料**: `d_henf` テーブルの `klsryo` カラムが保守料相当と推定する。Form_f_flx_KHIYOが同カラムを「保守料」として使用しているため。**確認必須**

2. **f_flx_TOUGETSU の請求月**: 集計月（DtFrom/DtToのパラメータ）ではなく現在月（CURRENT_DATE）を請求月として表示する可能性が高い。Form_f_flx_KEIJOが同様の実装をしているため。**確認推奨**

3. **f_flx_YOSAN の「予想」と「既存」**: 開始日が集計期間内にある（今後の予定）を「予想」、既に開始済みのものを「既存」と分類すると推定する。**確認必須**

4. **f_CHUKI_RECALC の gnzaiKt**: 現在価値（gnzaiKt）の計算式は `LEAST(b_knyukn, b_gnzai_kt)` （既存コードのb_syutokの計算式と同等）の可能性が高い。**確認必須**

5. **Form_BuknEntry の削除対象テーブル**: d_kykmレコードの削除時にd_haifも合わせて削除する必要があると推定する（外部キー制約により単独削除不可の可能性）。**確認推奨**

6. **f_flx_YOSAN のグレーアウト条件**: Form_f_flx_KHIYOの実装（期間外かつ計上額0の行）と同一条件と推定する。**確認推奨**
