## システム概要

### リースM4BSとは

リースM4BS（Lease Management for Business System）は、イルテックス社が開発したリース管理システムである。IFRSおよび日本基準に対応したオペレーティングリース・ファイナンスリースの契約管理・会計仕訳出力を業務範囲とする。

主な機能は以下のとおり。

- リース契約書の登録・管理（契約書ヘッダ・物件明細）
- 月次仕訳計上（計上仕訳・支払仕訳・経費仕訳）
- 返済スケジュール・償却スケジュールの自動計算
- 仕訳の外部システム（会計パッケージ）への出力
- 各種帳票・一覧表の出力
- マスタデータ（企業・口座・仕訳科目等）の保守

### 技術スタック

| 世代 | 実行環境 | 主要技術 | データベース |
|------|---------|---------|------------|
| Access版（現行） | Microsoft Access 2016 64bit Runtime | VBA / Access Form | Access MDB / Oracle |
| VB.NET版（移行中） | .NET Framework 4.7.2 | WinForms / VB.NET | PostgreSQL |

### GUI定義ファイルの構造

AccessGUIフォームの定義ファイルは以下の構造を持つ。

```text
Access Form定義言語 Version 21

Begin Form
    ...（フォームプロパティ）
    RecordSource = "tw_XXXX"     ' データソース（tw_プレフィックスのテーブル/ビュー）
    Begin Section
        ...（セクション定義）
        Begin Control
            ...（コントロール定義）
            OnOpen      = "[Event Procedure]"   ' VBAイベントハンドラ連携
            OnClose     = "[Event Procedure]"
        End
    End
    NameMap                       ' コントロール名のUnicode/16進エンコード
    PrtDevMode                    ' プリンター設定（16進数バイナリ）
End
```

主な特徴:

- `RecordSource` にはプレフィックス `tw_` のテーブルまたはビューを指定
- イベントハンドラ（`OnOpen`、`OnClose`、`OnClick` 等）は `[Event Procedure]` として宣言し、VBAモジュールと連携
- `NameMap` セクションでコントロール名をUnicodeまたは16進数でエンコード
- `PrtDevMode` セクションに16進数バイナリ形式のプリンター設定を格納

## システム起動フローとメニュー構成

### 起動処理の流れ

システム起動時に `p_StartUp_gMain()` が呼び出され、以下の順序で初期化処理が実行される。

```text
p_StartUp_gMain()
  ├─ Accessバージョン確認（2000～2016対応）
  ├─ 和暦配列 tgEras() 初期化（明治・大正・昭和・平成・令和）
  ├─ インストールフォルダチェック（ProgramFiles直下インストール時に警告）
  ├─ ACCESSウィンドウ最大化
  ├─ ワークDB初期化
  │   ├─ LM4BSwork.mdb       （ワーク用）
  │   ├─ LM4BSmdld.mdb       （モデル用）
  │   └─ LM4BSImpWkDB.mdb    （Excel取込用）
  ├─ カスタマイズ設定読込（T_Customize テーブル → igCUSTM_TYPE に格納）
  ├─ メニューバー生成（tcon_MenuMatrix テーブル → 5階層メニュー構築）
  ├─ セキュリティ設定（権限グループ別メニュー表示/非表示制御）
  └─ ログ初期化（Tempフォルダに LM42log.mdb を生成）
```

### ログイン方式

システムはデータベース接続方式によって2種類のログインモードを持つ。

| ログイン方式 | 接続先 | 概要 |
|-----------|-------|------|
| JETログイン | Access MDB | 標準構成。ローカルまたはファイルサーバー上のMDB直接接続 |
| Oracleログイン | Oracle DB | `T_Customize` の「サーバーDBはOracle」フラグが有効な場合に使用 |

ログイン情報は `typ_gLogin` 構造体で管理され、以下のフィールドを保持する。

| フィールド | 内容 |
|-----------|------|
| `UserCD` | ユーザーコード |
| `PWD` | パスワード（暗号化済み） |
| `KngnCd` | 権限グループコード |
| `boAdmin` | 管理者フラグ |
| `boMaster` | マスタ更新権限フラグ |
| `boAPPROVAL` | 承認権限フラグ |
| `boLOG` | ログ参照権限フラグ |
| `boFOutKNGN` | ファイル出力権限フラグ |
| `boPriKNGN` | 印刷権限フラグ |

ログイン試行は最大5回（`cn_gLOGIN_ATTEMPTS_SYS = 5`）に制限されている。

### メニュー構成（5階層、tcon_MenuMatrix）

メニューは `tcon_MenuMatrix` テーブルの定義に基づき、起動時に動的に生成される。最大5階層の階層メニューを構成し、ユーザーの権限グループ（`mMENUBAR_ENABLE_CTL`）によって表示・非表示が制御される。

メニューの大区分は以下のカテゴリで構成されている（フォーム名から推定）。

| カテゴリ | 主な機能 |
|---------|---------|
| 契約管理 | 契約書登録・参照・一覧、解約処理 |
| 物件管理 | 物件登録・参照・変更 |
| 月次処理 | 仕訳計上、支払処理 |
| 仕訳出力 | 計上仕訳・支払仕訳・経費仕訳の外部出力 |
| 帳票出力 | 棚卸・リース料・残高・債務等の帳票 |
| データ取込 | Excelインポート、各種データ取込 |
| マスタ管理 | 各種マスタの保守 |
| ツール | 祝日・消費税・契約番号シーケンス等 |
| システム | ユーザー管理・権限設定・ログ参照 |

### 画面遷移の全体像

起動後の基本的な画面遷移は以下のパターンに従う。

```text
[起動]
  └─ p_StartUp_gMain()
       └─ ログイン画面
            └─ メインメニュー（tcon_MenuMatrix 生成）
                 ├─ 条件入力画面（f_*_JOKEN）
                 │    └─ 一覧画面（f_flx_*）
                 │         └─ 明細入力画面（f_KYKH / f_KYKM 等）
                 ├─ 仕訳出力画面（f_仕訳出力標準_* / fc_*）
                 ├─ マスタ一覧（f_flx_M_*）
                 │    └─ マスタ入力（f_M_*_INP）
                 └─ システム設定（0F_SYSTEM等）
```

`f_KIRIKAE`（画面切替フォーム）がモーダルダイアログとして機能し、進捗表示には `f_StatusMeter` が使用される。

## リース計算ロジック

リースM4BSの計算エンジンはAccessVBA上に実装されており、IFRS16号・リース会計基準に対応した月次計上計算を行う。

### 月次仕訳計上フロー

月次計上処理のエントリポイントは `pc_月次仕訳計上` であり、以下の順で処理が実行される。

```text
pc_月次仕訳計上
└─ g月次_v計_契約書to料金TBL_計上()
    ├─ 対象期間の契約データ抽出（D_KYKH / D_KYKM）
    ├─ 契約ループ（各契約ごとに以下を実行）
    │  ├─ m月次_v計_SQLMAKE()       SQL生成（抽出条件組み立て）
    │  ├─ m月次_v計()               利息・償却の統合計算
    │  │  ├─ m月次_v計_SUB_償却関連()  償却関連サブ処理
    │  │  ├─ mMake償却_SCH()           償却スケジュール生成
    │  │  └─ gMake返済_SCH()           返済スケジュール生成
    │  └─ mMakeFUSAI_SCH()         債務スケジュール生成
    ├─ m利用_結果テーブル入力()     計算結果をtwテーブルへ書込
    └─ m登録処理()                  業務テーブルへ確定登録
```

### 年金現価計算

仕訳計上の基礎となるリース料の現在価値は年金現価公式で算出される。

- 計算定義: `0f_MNT_tcon_年金現価の計算式`
- 用途: リース開始時の資産計上額（使用権資産）および負債額（リース負債）の算定

```text
年金現価 = 定期支払額 × Σ [1 / (1 + r)^n]
           （r = 月利率, n = 残存期間月数）
```

追加借入利子率（仮利率）は `f_T_KARI_RITU` 画面で管理し、`T_SYSTEM` に保存される。

### 返済スケジュール生成（gMake返済_SCH）

各リース契約について、リース期間全体の返済計画を生成する。

| 列 | 内容 |
|----|------|
| 期日 | 各返済日（月次） |
| リース料 | 毎月の支払総額 |
| 元本返済額 | 負債元本の減少分 |
| 利息額 | 利息 = 期首残高 × 利率 / 12 |
| 期末残高 | 期首残高 - 元本返済額 |

### 償却計算（定額法 / 利息法）

`mMake償却_SCH` が以下2方式を切り替えて処理する。

| 方式 | 概要 | 適用場面 |
|------|------|---------|
| 定額法 | 資産残高 ÷ リース期間（月数）を毎月均等計上 | オペレーティングリース相当 |
| 利息法 | 期首帳簿価額 × 実効利子率で利息を計算し差額を元本返済 | ファイナンスリース（IFRS16対応） |

償却スケジュール結果は `tw_S_CHUKI_KEIJO`（注記計上）・`tw_D_HENL_KEIJO`（変更時）・`tw_D_GSON_KEIJO`（減損時）に格納される。

### 債務スケジュール生成（mMakeFUSAI_SCH）

リース負債の長短振替を含む債務スケジュールを生成する。

- **長期負債**: 決算日から1年超のリース負債残高
- **短期負債（流動負債）**: 決算日から1年以内に弁済予定の元本分
- 長短振替タイミングは会計年度終了月を基準に自動算出

### 変更・中途解約時の再計算

`f_HEN_SCH`（返済スケジュール変更）画面から変更が登録されると、変更日以降の償却・債務スケジュールが再計算される。変更前後の差額は `D_HENL`（変更履歴）に記録される。

## セキュリティモデル

リースM4BSは独自のユーザー認証・権限管理機構を持ち、Access JETセキュリティに依存しない独立したセキュリティレイヤーを実装している。

### ユーザー認証

ログイン情報は `typ_gLogin` 構造体としてメモリ上で保持される。

```vba
Type typ_gLogin
    UserCD      As String   ' ユーザーコード
    PWD         As String   ' パスワード（暗号化済み）
    KngnCd      As String   ' 権限グループコード
    boAdmin     As Boolean  ' 管理者フラグ
    boMaster    As Boolean  ' マスタ更新権限フラグ
    boAPPROVAL  As Boolean  ' 承認権限フラグ
    boLOG       As Boolean  ' ログ参照権限フラグ
    boFOutKNGN  As Boolean  ' ファイル出力権限フラグ
    boPriKNGN   As Boolean  ' 印刷権限フラグ
End Type
```

パスワードは `pc_Encrypt` 関数で暗号化して `SEC_USER` テーブルに保存される。

### パスワードポリシー

| 項目 | 内容 |
|------|------|
| ログイン試行制限 | 最大 5 回（`cn_gLOGIN_ATTEMPTS_SYS = 5`）超過でアカウントロック |
| パスワード有効期限 | T_SYSTEM で設定（日数指定） |
| 猶予期間 | 有効期限後も一定期間ログイン可能な猶予日数 |
| 最小文字数 | T_SYSTEM で設定 |
| 文字種チェック | 英字・数字・記号の混在要否を設定可能 |

### アクセス権限体系（ACCESS_KIND）

アクセス種別は3段階で管理される。

| ACCESS_KIND | 区分名 | 内容 |
|-------------|--------|------|
| 1 | 全データ更新 | 全契約管理単位・全部門のデータを更新可能 |
| 2 | 全データ参照 | 全データを参照のみ可能（更新不可） |
| 3 | 管理単位限定 | 自ユーザーに紐づく契約管理単位・部門のみアクセス可能 |

ACCESS_KIND=3（管理単位限定）の場合、以下のリストで制限範囲を管理する。

- `tgKKNRI_LIST`: 契約管理単位の許可リスト
- `tgBKNRI_LIST`: 部門管理単位の許可リスト

### 予約ユーザー

| ユーザーID | ユーザーコード | 用途 |
|-----------|--------------|------|
| 1 | system | システム管理者（全権限、顧客使用不可） |
| 0 | iltex | イルテックス保守専用アカウント |

### メニュー表示制御

権限グループに応じてメニューの表示/非表示を動的に制御する。

```text
mMENUBAR_ENABLE_CTL()
  ├─ tcon_MenuMatrix から権限グループ別メニュー定義を取得
  ├─ ACCESS_KIND・各boフラグを評価
  └─ 非許可メニュー項目をグレーアウト/非表示
```

### 操作ログ

| ログ種別 | テーブル | 記録内容 |
|---------|---------|---------|
| 操作ログ | `L_SLOG` | 画面開閉・主要ボタン操作の履歴 |
| 更新ログ | `L_ULOG` | データ変更前後の値（フィールド単位） |
| バックアップログ | `L_BKLOG` | バックアップ実行履歴 |

ログは Tempフォルダ配下の専用MDB（`LM42log.mdb`）に格納され、起動時に初期化される。ログ参照は `boLOG=True` の権限を持つユーザーのみ可能。

| 機能カテゴリ | 主なメソッド / 機能 |

### 今後の開発方針

1. **業務計算ロジックの先行移行**: 仕訳計上・スケジュール計算はシステムの根幹であり、早期に移行・テスト検証を完了させることが全体スケジュールの鍵となる。
2. **顧客固有実装の標準化**: 26 顧客分の `fc_` 系フォームを個別に移植するのではなく、仕訳出力の共通インターフェースを設計し、顧客ごとの差分を設定・プラグインとして管理する構造が望ましい。
3. **Access 版との並行稼働期間の設定**: 移行完了までの間、Access 版と VB.NET 版の計算結果を定期的に突き合わせ、ロジックの等価性を確認するテスト体制を整備する。
4. **固定長出力の早期完成**: KITOKU 顧客のような既存連携に依存している顧客は、`ToFixedLengthFile` の完成なしには VB.NET 版に切り替えられないため、優先度を上げて対処する。
