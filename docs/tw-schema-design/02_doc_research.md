# ドキュメント調査資料: tw-schema-design

## 1. 一時テーブル vs UNLOGGED TABLE vs 通常テーブル

### TEMPORARY TABLE
- セッション単位/トランザクション単位のスコープ
- 作成セッションからのみアクセス可能
- **カタログ肥大化が最大の問題**（頻繁な作成・削除でシステムカタログにメタデータ蓄積）
- Npgsqlのコネクションプーリングと相性が悪い（一時テーブルが残留するリスク）

### UNLOGGED TABLE
- WAL書き込みなし → 通常テーブルの約2-3倍高速
- 全セッションから共有アクセス可能
- クラッシュ時は自動TRUNCATE（データ消失）
- レプリカからアクセス不可、バックアップに含まれない

### 通常テーブル（本プロジェクトで採用）
- WAL書き込みあり、クラッシュリカバリ対応
- 既存コードとの一貫性を優先
- 将来の最適化としてUNLOGGED化を検討可能

## 2. スキーマ分離パターン

| 方式 | メリット | デメリット |
|------|----------|-----------|
| 専用スキーマ（`work`スキーマ） | 名前空間分離、GRANT制御 | 既存コード変更必要 |
| **テーブル接頭辞（`tw_`）** | **シンプル、既存命名規則と一致** | 同一スキーマ内で混在 |

**採用: `tw_` 接頭辞 + `public` スキーマ**（既存コードがスキーマ修飾なしでSQL発行）

## 3. TRUNCATE vs DELETE

| 項目 | TRUNCATE | DELETE |
|------|----------|--------|
| 計算量 | O(1) 定数時間 | O(n) 行数に比例 |
| トリガー | 発火しない | 各行でDELETEトリガー発火 |
| トランザクション | ロールバック可能 | ロールバック可能 |

**採用: DELETE FROM**（既存CrudHelperのトランザクション管理との互換性を優先）

## 4. インデックス設計

- B-tree（デフォルト）: 等価検索 + 範囲検索に対応 → tw_系のキーコード検索に最適
- Hash: 等価検索のみ → tw_系では不要
- 部分インデックス: フラグ列がある場合に検討

## 5. VB.NET + Npgsql パターン

- Npgsqlはデフォルトでコネクションプーリング有効
- CrudHelper が接続・トランザクション管理を統一
- DELETE FROM → INSERT パターンが確立済み
