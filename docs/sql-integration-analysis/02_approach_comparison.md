# アプローチ比較: SQL1/SQL2 統合方式の検討

## 前提

- SQL1（マイグレーション版）: 66テーブル + 約21ワークテーブル、日本語略称命名
- SQL2（新リース対応版）: 25テーブル、英語命名、ASBJ第16号対応
- 既存顧客が SQL1 ベースの仕訳パイプライン（KeijoSqlBuilder → tw_s_chuki_keijo → 仕訳出力）を本番運用中
- マスタデータの重複（m_bcat/m_department、m_lcpt/m_supplier、c_kkbn/m_contract_type 等）が発生

---

## アプローチ1: 3つの塊（SQL1 + SQL2 + SQL3中継テーブル）

### 概要

SQL1テーブルとSQL2テーブルをそのまま維持し、新たにSQL3として中継テーブル（ctb_*）を追加。SQL1 → SQL3 → SQL2 の方向でデータを同期する。

### 構成イメージ

```
[SQL1: 66+21テーブル]     [SQL3: 中継ctb_*]     [SQL2: 25テーブル]
  d_kykh ──────────────→ ctb_bridge_* ─────→ tw_lease_contract
  d_kykm ──────────────→ ctb_bridge_* ─────→ ctb_lease_integrated
  m_bcat ──(独立管理)──   ※マスタ非同期     ←── m_department
  m_lcpt ──(独立管理)──   ※マスタ非同期     ←── m_supplier
```

### メリット

- **既存影響ゼロ**: SQL1テーブルに一切の変更を加えない。既存の仕訳パイプライン（KeijoSqlBuilder等）は完全に動作を維持
- **段階移行可能**: SQL3中継テーブルを順次追加し、検証しながら進められる
- **リスク最小**: ロールバックは中継テーブルの削除のみで完了
- **新リース基準対応**: SQL2側のtw_lease_judgment / tw_lease_accounting は独立して開発可能

### デメリット

- **マスタ二重管理の恒常化**: m_bcat と m_department が永続的に並存。データ更新時の同期漏れリスクが常に存在
- **技術的負債の蓄積**: テーブル総数が86+（SQL1 87 + SQL2 25 + SQL3 中継）に膨張。新規開発者の学習コストが極めて高い
- **命名規則の混在**: 日本語略称（d_kykh）と英語（tw_lease_contract）が混在し続ける
- **パフォーマンス**: 中継テーブル経由のJOINが増加。特に仕訳出力時のクエリが複雑化

### 開発工数見積

| 作業 | 工数 |
|---|---|
| SQL3中継テーブル設計・実装 | 1-2週間 |
| 同期バッチ開発 | 1週間 |
| VB.NET側の接続統一 | 0.5週間 |
| テスト | 1週間 |
| **合計** | **3.5-4.5週間** |

## アプローチ2: 1つの塊（SQL4完全統合）

### 概要

SQL1とSQL2の全テーブルを英語命名で完全に再設計。統一されたスキーマで全機能を実現する。

### 構成イメージ

```
[SQL4: 統合スキーマ（推定50-60テーブル）]
  contracts          ← d_kykh 統合
  contract_items     ← d_kykm 統合
  dept_allocation    ← d_haif + ctb_dept_allocation 統合
  repayment_schedule ← d_henf + tw_lease_schedule 統合
  journal_work       ← tw_s_chuki_keijo + tw_lease_journal 統合
  departments        ← m_bcat + m_department 統合
  suppliers          ← m_lcpt + m_supplier 統合
  ...
```

### メリット

- **技術的負債ゼロリセット**: 命名規則統一、重複排除、正規化された理想的なスキーマ
- **保守性最高**: 新規開発者が理解しやすい。英語命名で国際化対応も可能
- **パフォーマンス最適**: 不要なJOINやビューが不要。最適なインデックス設計
- **新リース基準完全対応**: ASBJ第16号の要件を最初から組み込んだ設計

### デメリット

- **開発工数が現実的でない**: KeijoSqlBuilder（約250行）は d_kykh/d_kykm/d_haif を直接参照する動的SQL生成。全面書き換えが必要
- **ビッグバンリスク**: 全テーブルを一度に移行するため、部分的なロールバックが困難
- **仕訳パイプライン全面再構築**: tw_s_chuki_keijo → 仕訳出力標準_kj/sh/sm の全フローを再実装。tw_kitoku_* / tw_fc_swk_wrk の顧客固有出力も含む
- **既存顧客への影響**: 移行期間中、本番環境の仕訳出力が停止するリスク
- **データマイグレーション**: d_kykh（90カラム）、d_kykm（120カラム）のマッピング定義と検証が膨大

### 開発工数見積

| 作業 | 工数 |
|---|---|
| 統合スキーマ設計 | 2-3週間 |
| DDL実装 + マイグレーションスクリプト | 2週間 |
| KeijoSqlBuilder全面書き換え | 2-3週間 |
| 仕訳パイプライン再構築 | 3-4週間 |
| VB.NET DAL層全面改修 | 2-3週間 |
| テスト（仕訳出力一致検証含む） | 3-4週間 |
| **合計** | **14-19週間** |

## アプローチ3: ビュー統合 + 段階的マスタ統合（推奨）

### 概要

SQL1のテーブルをそのまま維持しつつ、SQL2のマスタを正（Single Source of Truth）として扱う。ビューとトリガーで双方向同期を実現し、段階的にSQL1テーブルをビュー化していく。

### 構成イメージ

```
Phase A: マスタ統合ビュー
  m_department（正）←→ v_m_bcat（ビュー + INSTEADOFトリガー）
  m_supplier（正）  ←→ v_m_lcpt（ビュー + INSTEADOFトリガー）
  m_contract_type（正）←→ v_c_kkbn（ビュー）

Phase B: 契約データ参照切替
  d_kykh / d_kykm    ← 既存のまま（KeijoSqlBuilder等が参照）
  ctb_lease_integrated ← migrate_d_kykh_to_ctb.sql で同期
  tw_lease_*          ← 新規リース判定・会計計算用

Phase C: 長期的ビュー化
  d_kykh → v_d_kykh（ctb_lease_integratedを参照するビュー）
  d_kykm → v_d_kykm（ctb_propertyを参照するビュー）
```

### Phase A: マスタ統合ビュー（1-2週間）

#### 対象

| SQL1（旧） | SQL2（新・正） | 実装方法 |
|---|---|---|
| m_bcat | m_department | m_bcatをビュー化。INSTEAD OFトリガーで書き込み対応 |
| m_lcpt | m_supplier | m_lcptをビュー化。INSTEAD OFトリガーで書き込み対応 |
| c_kkbn | m_contract_type | c_kkbnをビュー化（読み取り専用で十分） |
| m_corp | m_company | m_corpをビュー化 |
| m_koza | m_bank_account | m_kozaをビュー化 |
| sec_user | tm_USER | sec_userをビュー化（権限テーブルは別途検討） |

#### 効果

- マスタの二重管理を即座に解消
- 既存のSQL1参照コード（VB.NET）は変更不要（ビュー名 = 旧テーブル名）
- SQL2側からのマスタ更新がSQL1側に自動反映

### Phase B: 契約データ参照の段階的切り替え（2-4週間）

- `migrate_d_kykh_to_ctb.sql` の定期実行（バッチまたはトリガー）で d_kykh → ctb_lease_integrated の同期を維持
- 新規機能（リース判定 tw_lease_judgment、会計計算 tw_lease_accounting）は SQL2 テーブルを直接参照
- 既存の仕訳パイプライン（KeijoSqlBuilder → tw_s_chuki_keijo → 仕訳出力）は変更なし

### Phase C: 長期的にSQL1テーブルをビュー化

- d_kykh を ctb_lease_integrated のビューに段階的に置き換え
- KeijoSqlBuilder のSQL生成先をビュー経由に切り替え
- 最終的に物理テーブルはSQL2系のみとなる

### メリット

- **既存影響最小**: Phase Aのビュー化はSQL1参照コードの変更不要
- **マスタ二重管理の即時解消**: Phase Aだけで主要な重複が解消
- **段階的移行**: Phase B, C は独立してスケジュール可能。各Phaseで検証・ロールバック可能
- **仕訳パイプライン維持**: KeijoSqlBuilder等の既存コードに手を入れない
- **新リース基準対応**: tw_lease_* テーブルは独立して開発進行

### デメリット

- **トリガーのデバッグ難度**: INSTEAD OFトリガーの動作検証が複雑。特にマスタ更新時の副作用
- **ビュー増加**: 中間段階ではビューが多数存在し、スキーマが分かりにくくなる
- **Phase C の工数不確実性**: d_kykh（90カラム）のビュー化は、カラムマッピングが膨大

### 開発工数見積

| Phase | 作業 | 工数 |
|---|---|---|
| A | マスタ統合ビュー + トリガー | 1-2週間 |
| A | DB接続統一（init.sql統合） | 0.5週間 |
| A | テスト | 0.5週間 |
| B | migrate_d_kykh_to_ctb バッチ化 | 1週間 |
| B | 新機能（判定・会計）のSQL2直接参照実装 | 1-2週間 |
| B | テスト | 1週間 |
| C | d_kykh/d_kykm ビュー化設計 | 1-2週間 |
| C | KeijoSqlBuilder ビュー対応 | 1-2週間 |
| C | テスト（仕訳出力一致検証） | 1-2週間 |
| **合計** | | **Phase A: 2-3週間 / Phase B: 3-4週間 / Phase C: 3-6週間** |

| **アプローチ3（ビュー統合）** | **26/35** | **推奨** | バランス最良。Phase A だけでも効果大 |

### 推奨理由

アプローチ3を推奨する主な理由:

1. **Phase A の即効性**: マスタ統合ビューの導入（1-2週間）だけで、最も深刻な問題（マスタ二重管理）が解消される
2. **段階的リスク管理**: 各Phase を独立してスケジュール・検証可能。Phase B 以降は優先度に応じて後回しにもできる
3. **既存コード保護**: KeijoSqlBuilder（約250行の動的SQL）を含む仕訳パイプラインに手を入れないため、既存顧客の仕訳出力に影響しない
4. **DB接続の統一**: Phase A で init.sql を統合し、単一DB（lease_m4bs_dev）に SQL1 + SQL2 テーブルを同居させることで、DbConnectionManager.vb の変更が不要
