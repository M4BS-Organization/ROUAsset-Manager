## 1. エグゼクティブサマリー

### 実装概要

Issue #26 は、Access版に存在する fc_ プレフィックス付き52画面の仕訳出力フォームを VB.NET に移植するための**共通基盤の設計・実装**を目的としたタスクである。PR #85 で以下の成果物が納品された:

- **FcJournalOutputBase** — Template Method パターンによる仕訳出力基底クラス
- **FcSetteiHelper** — 顧客別設定の永続化ヘルパー（t_settei テーブル経由）
- **FixedLengthFileWriter** — Shift-JIS 固定長ファイル出力モジュール
- **KitokuFixedLengthFormats** — KITOKU向け4ワークテーブルのフィールド定義（52+72+34+63フィールド）
- **Form_fc_支払仕訳_KITOKU** — 支払仕訳出力画面（4ファイル出力対応）
- **Form_fc_計上仕訳_KITOKU** — 計上仕訳出力画面（1ファイル出力）
- **Form_fc_支払仕訳_KITOKU_SUB** — 消費税率・税処理コード入力サブフォーム
- **SQLテーブル定義** — tw_kitoku_cmsw2wrk / apgdhwrk / apgddwrk / apgdswrk

### 主要な達成事項

1. **共通基底クラスの設計完了** — FcJournalOutputBase による Template Method パターンが確立され、他顧客への展開基盤が整った
2. **設定永続化の仕組み完成** — FcSetteiHelper による顧客別設定の UPSERT / 読み込みが動作
3. **固定長ファイル出力の汎用モジュール完成** — FixedLengthFileWriter は Shift-JIS バイト単位パディング対応で、任意のフォーマット定義を受け取れる汎用設計
4. **KITOKU向けUI再現率 83.3%** — 支払仕訳・計上仕訳フォームのUI要素は概ね再現済み
5. **KITOKU向けフォーマット定義完成** — 4ワークテーブル合計221フィールドの固定長フォーマット定義がAccess版準拠で実装済み

### 主要な未達事項（致命的Gap）

1. **支払仕訳の仕訳分岐ロジック完全欠落（Gap B1: 致命的）** — Access版は売買/賃貸、現預金/未払金、消費税一括有無の6パターン分岐を持つが、VB.NET版は単一SQLのみ。正しい仕訳データが生成できない
2. **計上仕訳のデータ作成ロジック完全欠落（Gap B2: 致命的）** — Access版は開始計上（4サブ関数）、税一括（2サブ関数）、減価償却の3大分類を持つが、VB.NET版は単一SQLのみ。リース資産/債務計上、消費税、減価償却が正しく処理されない
3. **SQL列マッピング未検証（Gap D2: 高）** — 支払仕訳・計上仕訳ともにTODOコメントが残存し、Access版との照合が未完了
4. **APGD系ワークテーブルの列充足率が極端に低い（Gap C3: 高）** — 72列中3列、34列中6列、63列中4列のみINSERT。後続システム連携に支障
5. **2顧客動作の完了条件未達（Gap H1: 高）** — KITOKU 1顧客のみで、かつ基底クラスの Execute() を使わない独自フロー

## 3. 致命的・高深刻度Gapの詳細説明

### Gap B1（致命的）: 支払仕訳の6種仕訳サブ関数が未実装

**何が欠けているか:**
Access版の `m仕訳()` は取引形態と支払タイミングに応じて6パターンに分岐する:
1. `m仕訳_売買現預金()` — 売買処理（計上月=支払月）
2. `m仕訳_売買未払金()` — 売買処理（計上月<支払月）
3. `m仕訳_賃貸消費税一括有現預金()` — 賃貸借・消費税一括有（計上月=支払月）
4. `m仕訳_賃貸消費税一括有未払金()` — 賃貸借・消費税一括有（計上月<支払月）
5. `m仕訳_賃貸消費税一括無現預金()` — 賃貸借・消費税一括無（計上月=支払月）
6. `m仕訳_賃貸消費税一括無未払金()` — 賃貸借・消費税一括無（計上月<支払月）

VB.NET版は `BuildInsertToWrkSql` で単一の UNION ALL SQL のみ実装しており、上記分岐が完全に欠落している。

**業務上の影響:** 売買取引と賃貸借取引の仕訳パターンが区別されず、消費税一括処理の有無による金額計算も正しく行われない。業務上最も重要な仕訳生成ロジックが機能しないため、出力データは使用不可。

**対応工数:** 大（Access版の6関数を精査し、VB.NET側でSQL分岐またはコードロジックとして再実装が必要）

**対応方針:** Access版の各サブ関数のSQL/ロジックを個別に解析し、`BuildInsertToWrkSql` を分岐対応に拡張するか、独立した仕訳作成メソッドとして実装する。

---

### Gap B2（致命的）: 計上仕訳の3種データ作成関数が未実装

**何が欠けているか:**
Access版の `m仕訳データ作成()` は3大分類を処理する:
1. `m仕訳データ作成_開始計上()` — リース資産/リース債務 + 仮払消費税/リース未払金（サブ関数4つ: 1D/1C/2D/2C）
2. `m仕訳データ作成_税一括()` — 消費税税一括処理（サブ関数2つ: D/C）
3. `m仕訳データ作成_減価償却()` — 減価償却費/減価償却累計額

VB.NET版は単一の UNION ALL SQL のみで、上記3分類が完全に欠落。

**業務上の影響:** リース開始時の資産/債務の複式仕訳、消費税一括処理、減価償却費の計上が全て正しく行われない。計上仕訳としての機能を果たさない。

**対応工数:** 大（Access版の3大分類+6サブ関数の解析・再実装が必要）

**対応方針:** Gap B1と同様、Access版のロジックを個別解析し、VB.NET側で忠実に再実装する。開始計上の4サブ関数は借方/貸方の科目・金額マッピングが異なるため、個別SQLまたはメソッドとして実装が望ましい。

---

### Gap D2（高）: SQL列マッピングが未検証

**何が欠けているか:**
`BuildInsertToWrkSql` 内のSQL列マッピングについて、以下のTODOが残存:
- 支払仕訳 L117: 「Access版 m仕訳データ作成（支払仕訳）と照合して正確な列マッピングに更新する」
- 計上仕訳 L77: 「Access版 m仕訳データ作成（計上仕訳）と照合して正確な列マッピングに更新する」

**業務上の影響:** 科目コード、金額、税額等の列が正しい位置にマッピングされていない可能性がある。固定長出力ファイルのデータ破損に直結し、後続システムへの連携が失敗する。

**対応工数:** 中（Access版SQLとVB.NET版SQLの列単位での突合作業）

**対応方針:** Access版の各m仕訳関数内のSQLを抽出し、INSERT先の列名とSELECT元の列名を1対1で突合する。差異があれば VB.NET版を修正。

---

### Gap C3（高）: APGD系ワークテーブルの列充足率が極端に低い

**何が欠けているか:**
| テーブル | 全列数 | INSERT列数 | 充足率 |
|---|---|---|---|
| tw_kitoku_apgdhwrk | 72列 | 3列 | 4.2% |
| tw_kitoku_apgddwrk | 34列 | 6列 | 17.6% |
| tw_kitoku_apgdswrk | 63列 | 4列 | 6.3% |

**業務上の影響:** 固定長出力ファイルの大半のフィールドが空白またはゼロになる。後続システム（AP支払管理等）が必要とする振込先情報、手形情報、予備フィールド等のデータが全て欠落する。

**対応工数:** 中（Access版の `gAPGDHWRK作成`, `gAPGDDWRK作成`, `gAPGDSWRK作成` のSQLを解析し、VB.NET版のBuild系メソッドを拡充）

**対応方針:** Access版の pc_仕訳出力 クラス内の各出力メソッドからINSERT SQLを抽出し、不足列を追加する。

---

### Gap C1（高）: 計上仕訳のAPGD系ファイル未出力

**何が欠けているか:**
Access版の計上仕訳は CMSW2WRK + APGDHWRK + APGDDWRK + APGDSWRK の4ファイルを出力するが、VB.NET版は CMSW2WRK の1ファイルのみ。

**業務上の影響:** 計上仕訳のAPGD系ファイルが生成されないため、後続システム（AP支払管理等）へのデータ連携が行われない。

**対応工数:** 中（支払仕訳側の実装パターンを流用可能）

**対応方針:** Form_fc_支払仕訳_KITOKU の `BuildApgdhWrk` / `BuildApgddWrk` / `BuildApgdsWrk` と同等のメソッドを計上仕訳側にも実装する。

---

### Gap G1（高）: サブフォーム呼び出し未実装

**何が欠けているか:**
`Form_fc_支払仕訳_KITOKU_SUB`（消費税率・税処理コード入力）は実装済みだが、親フォーム（`Form_fc_支払仕訳_KITOKU`）から表示するためのボタンやトリガーが存在しない。

**業務上の影響:** ユーザーが消費税率・税処理コードを設定する手段がなく、仕訳作成時にこれらの値がデフォルトまたは未設定のまま処理される。税額計算の正確性に影響。

**対応工数:** 小（親フォームにボタン追加 + ShowDialog 呼び出しコード追加）

**対応方針:** 親フォームに「税処理設定」ボタンを追加し、`Form_fc_支払仕訳_KITOKU_SUB.ShowDialog()` で呼び出し、結果の `Zritu` / `ZeiCd` プロパティを取得して仕訳処理に反映する。

---

### Gap E1（高）: データ前処理チェック（m仕訳作成元データCHK）の欠如

**何が欠けているか:**
Access版では仕訳実行前に以下のチェックを実施:
1. 資産科目マスタの費目決定要素CD未入力チェック
2. HREL（費目関連テーブル）との結合チェック
3. 科目コード未設定データの有無確認

VB.NET版にはこれらのチェックが一切存在しない。

**業務上の影響:** マスタデータに不備があっても仕訳が実行され、不正な仕訳データが出力される。エラー原因の特定が困難になる。

**対応工数:** 中（Access版のチェックSQLを解析し、VB.NET側で同等のバリデーションを実装）

**対応方針:** `ExecuteKitoku` / `ExecuteKeijo` の冒頭にバリデーションメソッドを追加し、チェック不合格時はエラーメッセージを表示して処理を中断する。

---

### Gap F2（高）: 伝票番号採番方式の差異

**何が欠けているか:**
Access版は `tw_s_joken` またはシーケンステーブルから NEXTVAL 方式で伝票番号を自動採番する。VB.NET版は `txt_SLIP_NO_START_VAL` のデフォルト値 "1" を使用し、RowNumber ベースの連番生成。

**業務上の影響:** 既存運用で管理されている伝票番号との連番が断絶し、重複伝票番号が発生するリスクがある。

**対応工数:** 中（採番テーブルの設計・実装 + Form_Load での NEXTVAL 取得）

**対応方針:** PostgreSQL のシーケンスまたは採番管理テーブルを導入し、Form_Load で次番号を自動取得する仕組みに変更する。

---

### Gap B3（高）: pc_仕訳出力 共通クラスが未実装

**何が欠けているか:**
Access版の `pc_仕訳出力` クラスが提供する `gDivKMK_CD()`, `gOutputFile_Check()`, `gAPGDHWRK出力()`, `gAPGDDWRK出力()`, `gAPGDSWRK出力()` がVB.NET側に共通モジュールとして存在しない。

**業務上の影響:** 他顧客実装時に同じ出力ロジックを毎回個別実装する必要があり、共通基盤の設計意図に反する。

**対応工数:** 中（KITOKU内のインライン実装を共通クラスに抽出）

**対応方針:** `JournalOutputHelper` 等の共通クラスを新設し、科目コード分割・ファイルチェック・APGD系出力の共通メソッドを集約する。

---

### Gap B4（高）: 科目コード分割ロジック未実装

**何が欠けているか:**
Access版の `gDivKMK_CD` は `4160-001` 形式の科目コードを KMK_CD=`4160` / HKM_CD=`001` に分割する共通関数。VB.NET版には同等の処理がない。

**業務上の影響:** 科目コードが正しくマッピングされず、固定長出力の KMK_CODE / HKM_CODE 列に不正値が設定される可能性。

**対応工数:** 小（文字列分割の共通関数1つ）

**対応方針:** Gap B3 の共通クラス内に `DivideKamokuCode(code As String) As (KmkCd As String, HkmCd As String)` を実装する。
