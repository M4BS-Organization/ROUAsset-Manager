# Gap分析レポート — Access版 vs VB.NET実装（Issue #26）

調査日: 2026-03-19

---

## A. UI要素のGap

| Gap ID | カテゴリ | 深刻度 | 内容 | 影響 |
|--------|----------|--------|------|------|
| A1 | UI | 低 | **支払仕訳_KITOKU: txt_BSHO_CD のラベル差異** — Access版は「補助科目コード」、VB.NET版は「部署コード(一括)」と表記。Access版の `txt_BSHO_CD` は補助科目コードだが、VB.NET版では部署コードとして扱われている。 | ユーザーが入力項目の意味を誤解する可能性。実際の業務ロジック上の影響は列マッピング次第。 |
| A2 | UI | 低 | **計上仕訳_KITOKU: 出力ファイル名テキストボックスが1つのみ** — Access版は4ファイル出力だが、VB.NET版のUIは `txt_OUTPUT_FILE_NM` 1つのみ。ただしAccess版の計上仕訳もCMSW2WRKが主出力であり、APGD系はpc_仕訳出力クラス経由で共通処理のため、UIとしては妥当な可能性もある。 | 計上仕訳でAPGD系ファイルを出力する場合、ファイル名をUIから指定できない。 |
| A3 | UI | 中 | **支払仕訳_KITOKU_SUB: 親フォームからの呼び出しUIが未実装** — Access版では親フォーム内にサブフォームとして埋め込まれているが、VB.NET版では親フォーム（Form_fc_支払仕訳_KITOKU）にSUBフォームを表示するボタンやトリガーが存在しない。 | 消費税率・税処理コードの入力手段がユーザーに提供されない。 |

## B. 機能・ビジネスロジックのGap

| Gap ID | カテゴリ | 深刻度 | 内容 | 影響 |
|--------|----------|--------|------|------|
| B1 | 機能 | **致命的** | **支払仕訳: 6種の仕訳サブ関数が未実装** — Access版は `m仕訳()` 内で売買/賃貸、現預金/未払金、消費税一括有無の6パターンに分岐する仕訳作成ロジック（`m仕訳_売買現預金`, `m仕訳_売買未払金`, `m仕訳_賃貸消費税一括有現預金`, `m仕訳_賃貸消費税一括有未払金`, `m仕訳_賃貸消費税一括無現預金`, `m仕訳_賃貸消費税一括無未払金`）を持つ。VB.NET版は `BuildInsertToWrkSql` で単一のUNION ALL SQLのみ。 | 売買取引や消費税一括処理の区分に応じた仕訳パターンが正しく生成されない可能性が高い。業務上最も重要なロジックの欠落。 |
| B2 | 機能 | **致命的** | **計上仕訳: 3種の仕訳データ作成関数が未実装** — Access版は `m仕訳データ作成_開始計上`（4サブ: 1D/1C/2D/2C）、`m仕訳データ作成_税一括`（D/C）、`m仕訳データ作成_減価償却` の3大分類を持つ。VB.NET版は `BuildInsertToWrkSql` で単一SQLのみ。 | リース開始時の資産/債務計上、仮払消費税/リース未払金、減価償却費の仕訳が正しく生成されない。 |
| B3 | 機能 | 高 | **pc_仕訳出力 共通クラスが未実装** — Access版では `pc_仕訳出力`（`om仕訳出力`）クラスが `gDivKMK_CD()`, `gOutputFile_Check()`, `gAPGDHWRK出力()`, `gAPGDDWRK出力()`, `gAPGDSWRK出力()` を提供。VB.NET版ではこれらの共通メソッドが存在せず、KITOKUフォーム内にインライン実装。 | 他顧客（MYCOM, TSYSCOM等）の実装時に同じAPGD出力ロジックを毎回個別実装する必要がある。共通基盤の設計意図に反する。 |
| B4 | 機能 | 高 | **科目コード分割ロジック（gDivKMK_CD）が未実装** — Access版では `4160-001` 形式を KMK_CD=`4160`, HKM_CD=`001` に分割する共通関数がある。VB.NET版ではこの分割処理が見当たらない。 | 科目コードが正しくマッピングされない可能性。固定長出力でKMK_CODE/HKM_CODE列が不正値になる。 |
| B5 | 機能 | 中 | **基底クラスの Execute() が KITOKU から未使用** — `FcJournalOutputBase.Execute()` は `tw_fc_swk_wrk` を使う Template Method だが、KITOKU は `tw_kitoku_cmsw2wrk` を使う独自フロー（`ExecuteKitoku`/`ExecuteKeijo`）を採用。基底クラスの抽象化が形骸化。 | 他顧客実装時に基底クラスを使うか独自フローにするか判断基準が不明確。保守性の問題。 |

## C. ファイル出力のGap

| Gap ID | カテゴリ | 深刻度 | 内容 | 影響 |
|--------|----------|--------|------|------|
| C1 | ファイル出力 | 高 | **計上仕訳の出力ファイル数不足** — Access版は計上仕訳でも4ファイル（CMSW2WRK + APGDHWRK + APGDDWRK + APGDSWRK）を出力する。VB.NET版の計上仕訳は CMSW2WRK の1ファイルのみ出力。 | 計上仕訳のAPGD系ファイルが生成されないため、後続システム（AP支払管理等）にデータが連携されない。 |
| C2 | ファイル出力 | 中 | **ファイル拡張子の差異** — Access版は `.tmp` 拡張子、VB.NET版は `.txt` 拡張子で出力。 | 後続システムが `.tmp` を期待している場合、ファイル取り込みに失敗する。設定で変更可能なら影響は低い。 |
| C3 | ファイル出力 | 高 | **APGD系ワークテーブルの列充足率が極端に低い** — `BuildApgdhWrk` は72列中3列、`BuildApgddWrk` は34列中6列、`BuildApgdsWrk` は63列中4列のみ INSERT。残りはNULL/デフォルト。 | 固定長出力ファイルの大半のフィールドが空白/ゼロになり、後続システムが必要とするデータが欠落。 |
| C4 | ファイル出力 | 低 | **フィールド数のコメント不一致** — APGDDWRK: コメント「31フィールド」→実際34、APGDSWRK: コメント「62フィールド」→実際63。 | 保守時の混乱。機能上の直接的影響は少ない。 |

## D. DB処理・SQLのGap

| Gap ID | カテゴリ | 深刻度 | 内容 | 影響 |
|--------|----------|--------|------|------|
| D1 | DB | 中 | **ワークテーブル設計の二重構造** — 基底クラスは `tw_fc_swk_wrk`（汎用）を想定するが、KITOKU は `tw_kitoku_cmsw2wrk` 等の固有テーブルを使用。`tw_fc_swk_wrk` は実質未使用。 | テーブル管理の複雑化。将来の顧客追加時にどちらのテーブルを使うか混乱する。 |
| D2 | DB | 高 | **SQLの列マッピングが未検証（TODO状態）** — `BuildInsertToWrkSql` 内のSQL列マッピングについて、支払仕訳(L117)・計上仕訳(L77)ともに「Access版と照合して正確な列マッピングに更新する」というTODOが残存。 | 仕訳データの各列（科目コード、金額、税額等）が正しい位置にマッピングされていない可能性。出力ファイルのデータ破損に直結。 |
| D3 | DB | 中 | **SQLパラメータ化の不足** — `BuildInsertToWrkSql` 内で `slipDt`, `kamokuCd`, `bshoCd`, `slipNoStart` が文字列補間で直接SQL埋め込み。`NpgsqlParameter` 未使用。 | 内部用途のためSQLインジェクションリスクは低いが、特殊文字を含む入力でSQLエラーが発生する可能性。コーディング規約違反。 |

## E. エラー処理・バリデーションのGap

| Gap ID | カテゴリ | 深刻度 | 内容 | 影響 |
|--------|----------|--------|------|------|
| E1 | バリデーション | 高 | **m仕訳作成元データCHK() 相当処理の欠如** — Access版では実行前に資産科目マスタの費目決定要素CD未入力チェック、HREL（費目関連テーブル）との結合チェック、科目コード未設定データの有無確認を実施。VB.NET版にはこれらのデータ整合性チェックが存在しない。 | マスタデータ不備のまま仕訳作成が実行され、不正な仕訳データが出力される。エラー原因の特定が困難になる。 |
| E2 | バリデーション | 中 | **gOutputFile_Check() 相当処理の簡略化** — Access版の `pc_仕訳出力.gOutputFile_Check()` は出力ファイルパスの詳細な確認を行う。VB.NET版は `ValidateOutputFolder` でフォルダ存在チェックのみ。ファイル名の妥当性、既存ファイルの上書き確認等が不足。 | 不正なファイル名指定時や既存ファイル上書き時にユーザーへの警告がない。 |
| E3 | バリデーション | 中 | **txt_ZEI_CD の gLenCheck() バリデーション未実装** — Access版のサブフォームでは税処理コードに `gLenCheck()` 関数バリデーションが適用されている。VB.NET版の `Form_fc_支払仕訳_KITOKU_SUB` にはこのバリデーションがない。 | 不正な桁数の税処理コードが入力可能。 |

## F. 設定管理のGap

| Gap ID | カテゴリ | 深刻度 | 内容 | 影響 |
|--------|----------|--------|------|------|
| F1 | 設定 | 中 | **設定テーブルの差異** — Access版は `tw_s_joken` でフォームごとの設定を管理。VB.NET版は `t_settei` を使用し、`FcSetteiHelper` で `FC_{CustomerCode}_` プレフィックス付きキーで管理。テーブル構造・キー設計が異なる。 | Access版からの設定データ移行が必要。移行なしでは初期値がデフォルトに戻る。 |
| F2 | 設定 | 高 | **伝票番号採番方式の差異** — Access版は `NEXTVAL` 方式（シーケンス/採番テーブル使用）で伝票番号を自動採番。VB.NET版は `txt_SLIP_NO_START_VAL` のデフォルト値 "1" を使用し、RowNumber ベースの連番生成。 | 既存運用で連番管理されている伝票番号との整合性が取れない。重複伝票番号の発生リスク。 |
| F3 | 設定 | 中 | **設定駆動による出力形式切り替え未実装** — Issue #26 の完了条件「設定駆動による出力形式切り替えが動作」が部分達成。FcSetteiHelper は実装済みだが、出力形式（固定長/CSV/Excel）を設定で切り替えるロジックは未実装。 | 顧客ごとに異なる出力形式（TSYSCOM: CSV等）への対応ができない。 |

| H1 | スコープ | 高 | **「最低2顧客」の充足が曖昧** — 完了条件は「最低2顧客の固有仕訳出力が共通基盤上で動作」。現状 KITOKU 1顧客のみ実装済み（支払仕訳 + 計上仕訳の2画面）。「2顧客」を「KITOKU + 別顧客」と解釈する場合は未達。 | Issue #26 をクローズできない。他顧客（MYCOM, TSYSCOM等）はスケルトンのみで動作不可。 |

## サマリー

### 深刻度別集計

| 深刻度 | 件数 | Gap ID |
|--------|------|--------|
| 致命的 | 2 | B1, B2 |
| 高 | 9 | B3, B4, C1, C3, D2, E1, F2, G1, H1 |
| 中 | 9 | A3, B5, C2, D1, D3, E2, E3, F1, F3, H2, H3 |
| 低 | 3 | A1, A2, C4 |

### 最優先対応事項

1. **B1/B2（致命的）**: 支払仕訳の6種サブ関数・計上仕訳の3種データ作成関数が未実装。Access版の仕訳分岐ロジックが完全に欠落しており、正しい仕訳データが生成できない。
2. **D2（高）**: SQL列マッピングがTODO状態で未検証。出力データの正確性が保証されない。
3. **C1/C3（高）**: 計上仕訳のAPGD系ファイル未出力 + APGD系テーブルの列充足率が極端に低い。
4. **G1（高）**: SUBフォームの呼び出し未実装により消費税率設定が不可。
5. **E1（高）**: データ前処理チェック（m仕訳作成元データCHK相当）の欠如。
